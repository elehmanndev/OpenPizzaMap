# Part 2 — Security delta from today

## 2.1 — H2 resolved by deletion

`src/views/me.ejs` was rewritten end-to-end during the profile redesign. The submissions list (where `${s.rejectionReason}` was being interpolated unescaped) is gone. The new view escapes every interpolation through the `escapeHtml` / `escapeAttr` helpers wired into `app.locals` ([src/app.js:93-95](src/app.js)). Verified via grep: every `${...}` in the view is wrapped in one of the helpers ([src/views/me.ejs:19-105](src/views/me.ejs)).

**Status:** Closed. Re-open if `rejectionReason` (or any admin-authored free-text field) is rendered into a future view without escape.

## 2.2 — Session-cookie hardening (no CVE, but worth noting)

Before today, `app.set("trust proxy", ...)` was missing. Combined with `cookie.secure: true` in production, `express-session` was silently *refusing* to set the `opm.sid` cookie behind Hostinger's TLS-terminating proxy — every Google sign-in bounced back to `/auth`. Fixed in [`01accac`](https://github.com/ericll93/OpenPizzaMap/commit/01accac):

- `app.set("trust proxy", 1)` ([src/app.js:32-38](src/app.js)) so `req.secure` reflects the X-Forwarded-Proto header.
- `req.session.save(...)` in the OAuth callback ([src/routes/api.auth.js:57-63](src/routes/api.auth.js)) so the 302 fires only after PrismaSessionStore's MySQL write completes.

This was previously failing-closed (auth simply didn't work), so no historical exploit window. Going forward, sessions are correctly persisted with `Secure; HttpOnly; SameSite=Lax`.

## 2.3 — Magic-link removal

[`d918fc9`](https://github.com/ericll93/OpenPizzaMap/commit/d918fc9) removed:
- `POST /api/auth/start` (email + token issuance)
- `GET /verify` (token redemption)
- `GET /check-email` (post-issuance landing)
- `src/services/email.js` (SMTP transport)
- `test/auth-flow.test.js` (already stale)

Net effect: smaller attack surface. No more email-token forgery class, no SMTP credentials in prod env, no rate-limited email endpoint to abuse. The `verificationTokenHash` / `verificationTokenExpiresAt` columns on `User` are now orphaned but harmless — nullable, never read, never written. Drop in a follow-up cleanup migration when other schema changes ship; leaving them in place avoids a no-op migration that would re-tickle the Hostinger Prisma chmod issue.

## 2.4 — Avatar URL capture

`User.avatarUrl` ([prisma/schema.prisma:26](prisma/schema.prisma)) stores `profile.photos[0].value` from Google, validated through `pickGoogleAvatar()` (https + `*.googleusercontent.com`/`*.google.com` host allow-list) and sliced to 1024 chars ([src/services/googleAuth.js](src/services/googleAuth.js)). Rendered as `<img src="${escapeAttr(account.avatarUrl)}" referrerpolicy="no-referrer">` ([src/views/me.ejs:45](src/views/me.ejs)). See [N9](#n9--avatar-url-trust--resolved).

## 2.5 — C1, C2, N9 closed (2026-05-07 evening)

- **C1 — `prisma/dev.db` untracked.** `git rm --cached prisma/dev.db` plus `prisma/*.db` (and `*-journal`/`*-shm`/`*-wal`) added to `.gitignore`. The schema has been MySQL on prod for months; the SQLite file was a leftover from an earlier provider and has no consumer.
- **C2 — admin seed removed.** `prisma/seed.js` no longer creates an admin user. With Google-only auth, a row with a pre-baked `passwordHash` can never sign in anyway; admin promotion is now an explicit `UPDATE User SET role = 'admin' WHERE email = ...` against an already-signed-in Google user. The hardcoded password and the `console.log` that echoed it are both gone.
- **N9 — avatar URL allow-listed.** See [N9](#n9--avatar-url-trust--resolved).

# Part 3 — New findings

## N7 — `SESSION_SECRET` reused as DB password — **High**

- **Evidence:** Both `.env` and `.env.local` set `SESSION_SECRET="Cincdel42017."` and that same string is the password segment of `DATABASE_URL`. Identical values across two unrelated trust domains.
- **Impact:** A leak of one leaks the other. Session secret is what signs `opm.sid` cookies; if it leaks, an attacker can forge any user's session. DB password gives direct row access. They should fail independently.
- **Fix:** Generate a fresh session secret with `node -e "console.log(crypto.randomBytes(32).toString('hex'))"`, set it in both `.env` and Hostinger's `.builds/config/.env`, restart the worker. Existing logged-in users will be signed out once — acceptable given the user count (8 total).

## N8 — `GOOGLE_CLIENT_SECRET` exposed in chat history — **Medium**

- **Evidence:** During OAuth diagnosis I ran `grep -nE 'GOOGLE_CLIENT|...' .env` and the value `GOCSPX-…` was returned to the conversation. Also lives in `.env` on the local Windows box and on the Hostinger deploy. Per Google policy, any disclosed OAuth client secret should be rotated.
- **Impact:** Anyone with that secret + the public client ID can mint OAuth access tokens against this app's redirect URIs. Low practical impact (the redirect URIs are pinned to openpizzamap.com), but the policy treat-as-leaked is the right default.
- **Fix:** Google Cloud Console → APIs & Services → Credentials → reset the client secret. Update `.env`, `.env.local`, and Hostinger's `.builds/config/.env`. No code change needed. Document in [project memory](../.claude/projects/C--Users-Eric-OpenPizzaMap/memory) once done — same pattern as the existing Gemini-key-rotation entry.

## N9 — Avatar URL trust — **Resolved**

- **Original evidence:** `User.avatarUrl` was whatever Google returned in `profile.photos[0].value`, rendered with `escapeAttr` and `referrerpolicy="no-referrer"` but with no host allow-list.
- **Fix shipped:** `pickGoogleAvatar()` in [src/services/googleAuth.js](src/services/googleAuth.js) now parses the URL, requires `https:`, and requires the host to end with `.googleusercontent.com` or `.google.com`. Anything else returns `null` and the view falls back to the initials avatar. Still capped at 1024 chars.

# Part 4 — Operational notes (not security per se)

- The Hostinger Prisma chmod issue recurred during today's deploy and was resolved by SSH'ing in and running `npm run postinstall`. Path documented in [docs/runbook.md](docs/runbook.md) — `~/domains/openpizzamap.com/nodejs/`, with the `export PATH=/opt/alt/alt-nodejs20/root/bin:$PATH` requirement before `npm`. Suggested follow-up: move the postinstall sequence into `npm start` so every Passenger respawn self-heals without SSH.
- `.env` and `.env.local` contain plaintext production credentials on disk. They are correctly listed in `.gitignore` and have never been committed (verified via `git log --all -- .env .env.local`). This is fine for a single-developer local box, but is the kind of thing to revisit if a contributor is ever onboarded.
